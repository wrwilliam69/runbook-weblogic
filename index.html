WebLogic — Dominio · Paso 1
Variables de Entorno — Por qué y Para qué
Sin estas variables WebLogic no sabe dónde está Java ni sus propias librerías
⚠️ Problema sin variables

Si ejecutás wlst.sh sin las variables configuradas → error inmediato. El sistema no encuentra Java ni los JARs de WebLogic.

1Las 4 variables obligatorias
export JAVA_HOME=/u01/jdk-17.0.12
export MW_HOME=/u01/app/oracle/middleware
export WL_HOME=$MW_HOME/wlserver
export PATH=$JAVA_HOME/bin:$MW_HOME/oracle_common/common/bin:$PATH
VariableValorPor qué esa ruta
JAVA_HOME /u01/jdk-17.0.12 Le dice al SO dónde está Java instalado. Sin esto WLST no arranca porque no encuentra la JVM. Usamos JDK 17 porque WebLogic 14.1.2 requiere esta versión — JDK 11 es incompatible.
MW_HOME /u01/app/oracle/middleware MW = Middleware. Es la casa de toda la instalación Oracle/WebLogic. Aquí quedaron los binarios cuando hiciste el silent install. WLST lo usa para encontrar sus propias librerías.
WL_HOME /u01/app/oracle/middleware/wlserver La carpeta específica de WebLogic dentro del middleware. Contiene los templates JAR para crear dominios. Sin esto writeDomain() no encuentra la plantilla base.
PATH $JAVA_HOME/bin:...:$PATH Le dice al sistema dónde buscar ejecutables. Sin esto escribís java o wlst.sh y el sistema responde "comando no encontrado".
2Por qué usar export
ℹ️ Sin export vs Con export

Sin export: JAVA_HOME=/u01/jdk-17.0.12 → la variable solo existe en tu sesión. Los programas que lances NO la heredan.

Con export: todos los procesos hijos (como WLST) heredan la variable automáticamente.

Analogía: es una nota en tu escritorio. Sin export solo vos la ves. Con export todos los que trabajan desde tu escritorio también la ven.

⚠️ export solo dura la sesión SSH

Si cerrás la terminal → las variables desaparecen. Por eso se guardan en ~/.bashrc para que sean permanentes — ver sección siguiente.

3Verificar que quedaron bien
echo $JAVA_HOME   # debe mostrar /u01/jdk-17.0.12
echo $MW_HOME    # debe mostrar /u01/app/oracle/middleware
echo $WL_HOME    # debe mostrar /u01/app/oracle/middleware/wlserver
java -version    # debe mostrar openjdk version "17..."
⚠️ Linux es sensible a mayúsculas

$WL_HOME ≠ $Wl_HOME ≠ $wl_home. Siempre todo en mayúsculas. Error frecuente que hace que la variable aparezca vacía.

🧠 Las 4 variables — "JAVA-MW-WL-PATH"

JAVA_HOME → dónde está Java
MW_HOME → dónde está el Middleware (casa Oracle)
WL_HOME → dónde está WebLogic específicamente
PATH → para ejecutar java y wlst.sh sin ruta completa

.bashrc — Variables Permanentes
Para que las variables sobrevivan al cerrar la sesión SSH
1Diferencia entre archivos de configuración
ArchivoPara quéCuándo se lee
/etc/fstabMontaje de discosAl iniciar el sistema

ℹ️ Por qué ~/.bashrc y no /etc/.bashrc

~/.bashrc → solo tu usuario (ec2-user). El ~ es un atajo que significa el home del usuario actual: /home/ec2-user/.bashrc
/etc/bashrc → todos los usuarios del sistema

Para WebLogic usamos el del usuario porque el servicio corre como ec2-user.

2Agregar las variables al .bashrc
⚠️ Usar echo con doble mayor-que (»)

» agrega al final del archivo sin borrarlo. > (uno solo) sobrescribe y borra todo — nunca uses uno solo.

# Estos 4 comandos AGREGAN las líneas al .bashrc
echo 'export JAVA_HOME=/u01/jdk-17.0.12' >> ~/.bashrc
echo 'export MW_HOME=/u01/app/oracle/middleware' >> ~/.bashrc
echo 'export WL_HOME=/u01/app/oracle/middleware/wlserver' >> ~/.bashrc
echo 'export PATH=$JAVA_HOME/bin:$MW_HOME/oracle_common/common/bin:$PATH' >> ~/.bashrc
3Verificar y activar
# Ver las últimas 5 líneas del .bashrc para confirmar
tail -5 ~/.bashrc

# Aplicar los cambios sin cerrar la sesión
source ~/.bashrc

# Verificar que funcionan
echo $JAVA_HOME
echo $WL_HOME
ℹ️ Qué hace source

source ~/.bashrc recarga el archivo en la sesión actual sin necesidad de cerrar y abrir la terminal. Es como "aplicar cambios" en tiempo real.

Permisos Linux — Identificar y Resolver
Cómo saber cuándo un error es de permisos y qué hacer
1Cómo identificar que el error es de permisos
⚠️ Palabras clave en el mensaje de error

Cuando ves alguna de estas frases en un error → el problema es de permisos:
Permission denied · does not have write permission · cannot create · Access denied · Operation not permitted

# Error que apareció en WLST al crear el dominio:
# 60308: The domain location does not have write permission
# → ec2-user no puede escribir en /u01/domains/ porque el dueño era root
2Leer la salida de ls -la
ls -ld /u01/domains/
# Salida:   drwxr-xr-x. root root /u01/domains/
#           ↑           ↑    ↑
#           permisos    dueño grupo
ParteValorSignifica
drwxr-xr-xpermisosd=directorio · rwx=dueño puede leer/escribir/ejecutar · r-x=grupo y otros solo leen
rootusuario dueñoSolo root puede escribir en este directorio

1¿Por qué /etc/systemd/system/?
ℹ️ El registro oficial de servicios de Linux

/etc/systemd/system/ es donde viven todos los servicios del sistema: nginx, postgresql, sshd, etc. Es el "registro oficial" de Linux moderno (systemd).

Cuando ejecutás systemctl enable weblogic, Linux busca ahí el archivo weblogic.service.

Analogía AIX: Es el equivalente de configurar el arranque en /etc/rc.d o similar. Mismo concepto, diferente implementación.

2Crear el archivo de servicio

ℹ️ Cómo pegar en vi

1. Abrir el archivo con vi → 2. Presionar i → modo INSERT → 3. Clic derecho en terminal = pegar → 4. ESC → 5. :wq + Enter

⚠️ Por qué importa el orden

Borrar en orden incorrecto puede dejar recursos huérfanos que siguen generando costos aunque la instancia no exista. Por ejemplo: un Elastic IP sin instancia asociada cobra por hora. Los volúmenes EBS sin instancia también cobran.

1Orden correcto de eliminación
1

EC2 → Instances → seleccionar instancia → Instance state → Terminate instance. Diferencia importante: Stop = pausa (sigue cobrando almacenamiento) · Terminate = eliminación definitiva.
ℹ️ Stop vs Terminate

Stop → instancia apagada pero los volúmenes EBS persisten y cobran.
Terminate → instancia y sus volúmenes raíz se eliminan. Usar esto para limpiar el lab.

EC2 → Elastic Block Store → Volumes → filtrar por estado "available" (sin instancia asociada) → seleccionar → Actions → Delete volume.
⚠️ Volúmenes que pueden quedar

3

EC2 → Network & Security → Elastic IPs → seleccionar la IP → Actions → Release Elastic IP address. Una Elastic IP sin instancia asociada cobra ~$0.005 por hora.
4

EC2 → Security Groups → seleccionar el que creaste → Actions → Delete. No se puede eliminar si aún tiene instancias asociadas — por eso se hace al final.
5

Qué son, por qué importan y por qué systemd las necesita explícitamente
ℹ️ Por qué son necesarias

Las variables de entorno le dicen al sistema dónde están instaladas las herramientas. Sin ellas, ejecutar un script de WebLogic falla porque el sistema no sabe dónde está Java ni dónde está el dominio.

3Por qué systemd NO hereda estas variables
⚠️ Este es el error más confuso de systemd + WebLogic

Un script que funciona perfectamente desde tu terminal SSH falla cuando lo ejecuta systemd. La razón es que systemd arranca con un entorno completamente limpio.

# Terminal SSH (sesión interactiva):
# 1. Linux carga ~/.bash_profile automáticamente
# 2. Se definen: JAVA_HOME, MW_HOME, DOMAIN_HOME, WL_HOME, PATH
# 3. Ejecutas ./startNodeManager.sh → funciona ✅

# systemd (arranque automático):
# 1. Linux NO carga ningún .bash_profile
# 2. Entorno limpio: JAVA_HOME=?  MW_HOME=?  DOMAIN_HOME=?
# 3. Ejecutas startNodeManager.sh → FALLA — no encuentra Java ❌

# SOLUCIÓN: declarar Environment= en el archivo .service
ArchivoQué hace con las variables
~/.bash_profileSe carga en sesiones SSH interactivas. systemd NO lo lee.
/etc/systemd/system/*.serviceAquí se declaran las variables con Environment= porque .bash_profile no se carga.
$DOMAIN_HOME/bin/setDomainEnv.shScript de WebLogic que configura variables del dominio. Lo llaman internamente startWebLogic.sh y startNodeManager.sh — pero necesita que WL_HOME y DOMAIN_HOME ya existan.
ℹ️ Regla de oro

Todo lo que necesitas en ~/.bash_profile para que funcione en terminal, lo necesitas como Environment= en el archivo .service para que funcione con systemd.

4Archivo .service correcto con variables declaradas

[Install]

⚠️ Por qué /bin/bash es obligatorio en ExecStart

systemd por defecto usa /bin/sh (dash) para ejecutar scripts. Los scripts de WebLogic usan sintaxis bash que dash no interpreta. El servicio falla silenciosamente o con errores confusos.
Siempre: ExecStart=/bin/bash /ruta/script.sh
Nunca: ExecStart=/ruta/script.sh

Multi Datasource — Failover JDBC
Cómo WebLogic cambia automáticamente de conexión si la BD falla
1Qué es y para qué sirve
ℹ️ Analogía del cajero automático

Sin Multi Datasource: la aplicación tiene un solo cajero. Si se daña → sin servicio.
Con Multi Datasource: hay dos cajeros. Si el primero falla, el cliente pasa automáticamente al segundo sin notarlo. 

# Sin Multi Datasource (lo que teníamos):
Aplicación → jdbc/OracleXEDS → Oracle XE
                                    ↓ si Oracle cae
                               Aplicación cae ❌

# Con Multi Datasource (lo que configuramos):
Aplicación → jdbc/OracleMultiDS (el "paraguas")
                  ├── jdbc/OracleXEDS  → Oracle XE  (primario)
                  └── jdbc/OracleXEDS2 → Oracle XE  (fallback)
                                    ↓ si el primero falla
                               Cambia automáticamente al segundo ✅
ℹ️ En este lab los dos apuntan al mismo Oracle

Con un solo servidor Oracle, si Oracle cae ambos datasources fallan igual. El ejercicio sirve para aprender la configuración — en producción cada datasource apuntaría a un nodo Oracle diferente (Oracle RAC o Data Guard).

JDBC y Connection Pool
Cómo WebLogic conecta las aplicaciones con la base de datos Oracle
ℹ️ Qué es JDBC

JDBC (Java Database Connectivity) = API de Java para conectarse a bases de datos. WebLogic actúa como intermediario entre la aplicación y la BD — gestiona las conexiones de manera eficiente con un pool.

1Por qué un Connection Pool
⚠️ Sin pool — problema grave

Sin pool: cada request de usuario abre una conexión nueva a Oracle → lento y consume recursos. Con pool: WebLogic mantiene conexiones abiertas y las reutiliza → rápido y eficiente.

Sin PoolCon Pool (WebLogic DataSource)
1 usuario = 1 conexión nueva a Oracle100 usuarios comparten 10 conexiones
Tiempo de apertura en cada requestConexión ya lista — tiempo mínimo
Oracle se satura con muchas conexionesLímite controlado por WebLogic

ℹ️ Error de BD vs Error de WebLogic

Si el test de conexión en WebLogic falla, primero verifica con netstat/telnet si Oracle está escuchando. Si el puerto responde pero el DataSource falla → problema de credenciales o driver. Si el puerto no responde → Oracle no está corriendo.

🧠 Flujo de conexión JDBC

App Java pide conexión → JNDI lookup → WebLogic DataSource → Connection Pool → Oracle DB (1521)

Si falla: verificar en orden → ¿Oracle corre? (netstat) → ¿Puerto accesible? (telnet) → ¿Credenciales correctas? → ¿Driver correcto?

⚠️ Lección aprendida — No instalar Oracle y WebLogic en la misma instancia pequeña

En el laboratorio se intentó instalar Oracle XE en la misma EC2 que ya tenía WebLogic (t2.micro/t2.small). El resultado fue que el sistema se quedó sin RAM durante la instalación — el OOM Killer del kernel mató procesos para sobrevivir. La evidencia quedó en /var/log/messages:
Out of memory: Killed process 1676 (dnf)
Conclusión: una EC2 dedicada por servicio es la arquitectura correcta.

1Conceptos clave antes de configurar
ℹ️ JNDI — el sistema de nombres interno de WebLogic

Sistema de nombres interno de WebLogic que permite a las aplicaciones encontrar recursos (como conexiones a BD) por nombre, sin necesitar saber la IP, puerto ni contraseña directamente.

Analogía: Es como una agenda telefónica interna. La aplicación no llama a Oracle directamente — le pregunta a WebLogic: "dame lo que está registrado con el nombre jdbc/OracleXEDS". WebLogic busca en su agenda y entrega la conexión lista.

ℹ️ Pool de conexiones — por qué no conectar cada vez

Sin pool: cada request del usuario abre una conexión nueva a Oracle → lento, consume recursos, Oracle se satura.
Con pool (Datasource): WebLogic mantiene X conexiones abiertas y las reutiliza. 100 usuarios comparten 10 conexiones — más rápido, controlado.

Definición completa: "Un datasource JDBC es un objeto registrado en el árbol de nombres de WebLogic (JNDI) que provee conectividad a la base de datos a través de un pool de conexiones JDBC."

ℹ️ Por qué bind mount y no symlink

El instalador de Oracle verifica que /opt/oracle sea un directorio real dueño de oracle — no acepta symlinks. El bind mount hace que /opt/oracle y /u01/oracle sean el mismo lugar físico, pero el instalador lo ve como un directorio normal.

# Crear directorio real en /opt (no symlink)
sudo mkdir -p /opt/oracle
sudo chown oracle:oinstall /opt/oracle

# Montar /u01/oracle sobre /opt/oracle
# El instalador escribe en /opt/oracle pero los bytes van a /u01/oracle
sudo mount --bind /u01/oracle /opt/oracle

# Verificar
ls -la /opt/ | grep oracle
# Debe mostrar: drwxr-xr-x oracle oinstall oracle

# Verificar espacio disponible en /opt/oracle

# Debe mostrar el tamaño de /u01 (lv_domain), no el de /
⚠️ Errores frecuentes con symlinks vs bind mount

ln -s /u01/oracle /opt/oracle → Oracle rechaza symlinks
ln -s /opt/oracle /opt/oracle → symlink circular (apunta a sí mismo)
ln -s opt/oracle /opt/oracle → ruta relativa — falla si no estás en /
Siempre usar bind mount para Oracle.

Oracle XE necesita ~7 GB solo para binarios + datos iniciales. Si el volumen raíz tiene menos de ese espacio disponible, el configure falla. En AWS se puede ampliar el disco EBS sin apagar la instancia — pero el OS no se entera automáticamente, hay que decirle.

EBS permite modificar el tamaño en caliente. La instancia sigue corriendo mientras AWS amplía el volumen físico. Espera 1-2 minutos a que el estado cambie a "in-use - optimizing".

2Por qué necesitas dos comandos después
⚠️ AWS amplía el disco pero el OS no lo sabe todavía

El disco tiene 3 capas independientes. AWS solo amplía la capa más baja — las otras dos siguen viendo el tamaño viejo hasta que se les diga explícitamente que crezcan.

EBS Volume (hardware)      ← AWS lo amplió a 20GB ✅
    └── Partición xvda4    ← aún cree que es 15GB ❌
            └── Filesystem XFS  ← aún cree que es 15GB ❌
3Extender partición y filesystem
# PASO 1: extender la partición para que ocupe el nuevo espacio del disco
# growpart = herramienta que expande particiones
# /dev/xvda = el disco
# 4         = número de partición (xvda4 es donde vive /)
sudo growpart /dev/xvda 4
# Resultado esperado:
# CHANGED: partition=4 old: size=28946399 new: size=39432159

# PASO 2: extender el filesystem XFS para usar toda la partición
# xfs_growfs = herramienta para crecer filesystems XFS (RHEL 9 usa XFS por defecto)
# /          = punto de montaje a extender
sudo xfs_growfs /
# Resultado esperado:
# data blocks changed from 3618299 to 4929019

# PASO 3: verificar que el espacio nuevo está disponible
df -h /
# Debe mostrar el nuevo tamaño con más espacio libre

4Cuándo NO usar growpart
⚠️ Si el disco usa LVM (como /u01 en tu lab)

En este lab /u01 usa LVM. Para expandir un volumen LVM el proceso es diferente:
lvextend -L +5G /dev/vg_weblogic/lv_domain → extender el LV
xfs_growfs /u01 → extender el filesystem
growpart solo aplica para particiones directas (sin LVM), como el volumen raíz /.

🧠 Ampliar disco en AWS — "3 capas, 3 pasos"

AWS Console → amplía el volumen EBS (capa hardware)
growpart → amplía la partición (capa OS)
xfs_growfs → amplía el filesystem (capa datos)

Olvidar cualquiera de los tres pasos = el espacio nuevo no aparece en df -h.

⚠️ Estado D — señal de IO bound

Si ves varios procesos en estado D simultáneamente, es confirmación directa de IO bound. El disco está tan lento que los procesos no pueden continuar. Correlacionar con iostat -x 1.

4Ordenar por CPU y Memoria
Procesos que más CPU consumen
ps aux --sort=-%cpu | head
# --sort=-%cpu = ordena de MAYOR a MENOR por %CPU
# head = muestra solo los primeros 10
Procesos que más Memoria consumen
ps aux --sort=-%mem | head
# --sort=-%mem = ordena de MAYOR a MENOR por %MEM
# El signo - antes del campo = orden descendente (mayor primero)
ℹ️ El signo menos en --sort

En --sort el signo menos (–) antes del nombre del campo significa orden descendente — del mayor al menor. Sin el menos, ordena ascendente (menor primero, no útil para encontrar el culpable).

Reglas Mnemotécnicas
🧠 Primer paso — "CPUs Antes Que Nada"

Antes de interpretar cualquier load: nproc (Linux) o lparstat -i (AIX). Sin CPUs el load no significa nada.

🧠 3 tipos de falla — "C-M-I"

CPU Bound → id casi en cero (menor a 5%)
Memory Pressure → swap creciendo + id alto
IO Bound → wa alto (mayor a 10%) + id alto

🧠 Trampa de entrevista

Load alto NO siempre es CPU saturada. Si load alto PERO idle alto → el problema es IO o memoria.

🧠 AIX — 4 grandes cambios vs Linux

1. top → topas  |  2. systemctl → lssrc/startsrc/stopsrc
3. journalctl → errpt  |  4. df -h → df -g

🧠 SWAP — "FAL-CH-MK-ON"
