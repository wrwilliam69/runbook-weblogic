WebLogic — Dominio · Paso 1
Variables de Entorno — Por qué y Para qué
Sin estas variables WebLogic no sabe dónde está Java ni sus propias librerías
⚠️ Problema sin variables

Si ejecutás wlst.sh sin las variables configuradas → error inmediato. El sistema no encuentra Java ni los JARs de WebLogic.

1Las 4 variables obligatorias
export JAVA_HOME=/u01/jdk-17.0.12
export MW_HOME=/u01/app/oracle/middleware
export WL_HOME=$MW_HOME/wlserver
export PATH=$JAVA_HOME/bin:$MW_HOME/oracle_common/common/bin:$PATH
VariableValorPor qué esa ruta
JAVA_HOME /u01/jdk-17.0.12 Le dice al SO dónde está Java instalado. Sin esto WLST no arranca porque no encuentra la JVM. Usamos JDK 17 porque WebLogic 14.1.2 requiere esta versión — JDK 11 es incompatible.
MW_HOME /u01/app/oracle/middleware MW = Middleware. Es la casa de toda la instalación Oracle/WebLogic. Aquí quedaron los binarios cuando hiciste el silent install. WLST lo usa para encontrar sus propias librerías.
WL_HOME /u01/app/oracle/middleware/wlserver La carpeta específica de WebLogic dentro del middleware. Contiene los templates JAR para crear dominios. Sin esto writeDomain() no encuentra la plantilla base.
PATH $JAVA_HOME/bin:...:$PATH Le dice al sistema dónde buscar ejecutables. Sin esto escribís java o wlst.sh y el sistema responde "comando no encontrado".
2Por qué usar export
ℹ️ Sin export vs Con export

Sin export: JAVA_HOME=/u01/jdk-17.0.12 → la variable solo existe en tu sesión. Los programas que lances NO la heredan.

Con export: todos los procesos hijos (como WLST) heredan la variable automáticamente.

Analogía: es una nota en tu escritorio. Sin export solo vos la ves. Con export todos los que trabajan desde tu escritorio también la ven.

⚠️ export solo dura la sesión SSH

Si cerrás la terminal → las variables desaparecen. Por eso se guardan en ~/.bashrc para que sean permanentes — ver sección siguiente.

3Verificar que quedaron bien
echo $JAVA_HOME   # debe mostrar /u01/jdk-17.0.12
echo $MW_HOME    # debe mostrar /u01/app/oracle/middleware
echo $WL_HOME    # debe mostrar /u01/app/oracle/middleware/wlserver
java -version    # debe mostrar openjdk version "17..."
⚠️ Linux es sensible a mayúsculas

$WL_HOME ≠ $Wl_HOME ≠ $wl_home. Siempre todo en mayúsculas. Error frecuente que hace que la variable aparezca vacía.

🧠 Las 4 variables — "JAVA-MW-WL-PATH"

JAVA_HOME → dónde está Java
MW_HOME → dónde está el Middleware (casa Oracle)
WL_HOME → dónde está WebLogic específicamente
PATH → para ejecutar java y wlst.sh sin ruta completa

WebLogic — Dominio · Paso 2
.bashrc — Variables Permanentes
Para que las variables sobrevivan al cerrar la sesión SSH
1Diferencia entre archivos de configuración
ArchivoPara quéCuándo se lee
/etc/fstabMontaje de discosAl iniciar el sistema
~/.bashrcVariables de entorno del usuarioAl iniciar sesión SSH de ese usuario
/etc/profileVariables globales para todos los usuariosAl iniciar cualquier sesión
ℹ️ Por qué ~/.bashrc y no /etc/.bashrc

~/.bashrc → solo tu usuario (ec2-user). El ~ es un atajo que significa el home del usuario actual: /home/ec2-user/.bashrc
/etc/bashrc → todos los usuarios del sistema

Para WebLogic usamos el del usuario porque el servicio corre como ec2-user.

2Agregar las variables al .bashrc
⚠️ Usar echo con doble mayor-que (»)

» agrega al final del archivo sin borrarlo. > (uno solo) sobrescribe y borra todo — nunca uses uno solo.

# Estos 4 comandos AGREGAN las líneas al .bashrc
echo 'export JAVA_HOME=/u01/jdk-17.0.12' >> ~/.bashrc
echo 'export MW_HOME=/u01/app/oracle/middleware' >> ~/.bashrc
echo 'export WL_HOME=/u01/app/oracle/middleware/wlserver' >> ~/.bashrc
echo 'export PATH=$JAVA_HOME/bin:$MW_HOME/oracle_common/common/bin:$PATH' >> ~/.bashrc
3Verificar y activar
# Ver las últimas 5 líneas del .bashrc para confirmar
tail -5 ~/.bashrc

# Aplicar los cambios sin cerrar la sesión
source ~/.bashrc

# Verificar que funcionan
echo $JAVA_HOME
echo $WL_HOME
ℹ️ Qué hace source

source ~/.bashrc recarga el archivo en la sesión actual sin necesidad de cerrar y abrir la terminal. Es como "aplicar cambios" en tiempo real.

WebLogic — Dominio · Paso 3
WLST — Crear el Dominio WebLogic
WebLogic Scripting Tool: consola de comandos Python para administrar dominios sin GUI
1¿Qué es WLST?
ℹ️ WLST = WebLogic Scripting Tool

Es una consola de comandos de WebLogic que habla Python por dentro. Permite crear y administrar dominios sin interfaz gráfica.

Analogía: Es como la consola web de WebLogic pero en modo terminal. Todo lo que se puede hacer en la consola web, se puede hacer en WLST.

2Encontrar y ejecutar WLST
# Encontrar dónde está WLST
find /u01 -name "wlst.sh" 2>/dev/null
# Aparece en dos rutas — ambas son válidas, usamos la segunda

# Ejecutar WLST (funciona porque agregamos el PATH al .bashrc)
wlst.sh

# La primera vez tarda — escanea todos los JARs. Esperar prompt:
# wls:/offline>
ℹ️ offline en el prompt

wls:/offline> significa que estás dentro de WLST pero no conectado a ningún dominio. Desde aquí se crean dominios nuevos.

3Crear el dominio — paso a paso dentro de WLST
📌 Analogía

WLST es como un formulario de configuración en terminal. Cada comando llena un campo: contraseña, puerto, ruta de instalación.

1
Cargar la plantilla base del dominio
readTemplate('/u01/app/oracle/middleware/wlserver/common/templates/wls/wls.jar')
# El prompt cambia a: wls:/offline/base_domain>
# WARNING que aparece = normal, no es error
ℹ️ Por qué esa ruta

/wlserver/common/templates/wls/wls.jar es la plantilla base de Oracle que define la estructura mínima de un dominio WebLogic. Sin ella no hay dominio.

2
Configurar contraseña del administrador
cd('/Security/base_domain/User/weblogic')
cmo.setPassword('Weblogic1!')
# Navega al nodo de seguridad y define la contraseña del usuario weblogic
ℹ️ Requisitos de contraseña WebLogic

Mínimo 8 caracteres, debe incluir número y carácter especial. Weblogic1! cumple: mayúscula + número + signo.

3

cd('/Servers/AdminServer')
set('ListenPort', 7001)
# Puerto estándar de WebLogic. 7002 es HTTPS
# Prompt: wls:/offline/base_domain/Server/AdminServer>
4

setOption('OverwriteDomain', 'true')
writeDomain('/u01/domains/SmartVistaDomain')

# El nombre del dominio lo defines vos aquí
# Prompt: wls:/offline/SmartVistaDomain/Server/AdminServer>
ℹ️ Por qué /u01/domains/

Convención Oracle: /u01 es el volumen dedicado al middleware. Los dominios siempre van en /u01/domains/NombreDelDominio. En Credibanco el dominio se llamaba SmartVistaDomain y quedó como nombre definitivo aunque sea el nombre técnico.

5
Salir de WLST y verificar
exit()

# Verificar que el dominio fue creado
ls -lrt /u01/domains/
# Debe aparecer: drwxr-x--- SmartVistaDomain

WebLogic — Dominio · Permisos Linux
Permisos Linux — Identificar y Resolver
Cómo saber cuándo un error es de permisos y qué hacer
1Cómo identificar que el error es de permisos
⚠️ Palabras clave en el mensaje de error

Cuando ves alguna de estas frases en un error → el problema es de permisos:
Permission denied · does not have write permission · cannot create · Access denied · Operation not permitted

# Error que apareció en WLST al crear el dominio:
# 60308: The domain location does not have write permission
# → ec2-user no puede escribir en /u01/domains/ porque el dueño era root
2Leer la salida de ls -la
ls -ld /u01/domains/
# Salida:   drwxr-xr-x. root root /u01/domains/
#           ↑           ↑    ↑
#           permisos    dueño grupo
ParteValorSignifica
drwxr-xr-xpermisosd=directorio · rwx=dueño puede leer/escribir/ejecutar · r-x=grupo y otros solo leen
rootusuario dueñoSolo root puede escribir en este directorio
rootgrupo dueñoEl grupo root también tiene acceso
ℹ️ Por qué ec2-user no podía escribir

El directorio era de root. El usuario ec2-user cae en la categoría "otros" que solo tiene r-x (leer y ejecutar, pero NO escribir). Por eso WLST fallaba al intentar crear el dominio.

3Resolver con chown
# chown = change owner (cambiar dueño)
# -R = recursivo (aplica a todo lo que hay dentro)
# ec2-user:ec2-user = nuevo usuario:nuevo grupo
sudo chown -R ec2-user:ec2-user /u01/domains/

# Verificar que cambió
ls -ld /u01/domains/
# Ahora debe mostrar: drwxr-xr-x. ec2-user ec2-user
4¿Cuándo usar sudo y cuándo no?
AcciónUsuario correctoPor qué
Instalar software en /etc, /usrsudoSon directorios del sistema, solo root puede modificarlos
Crear archivos en /u01/ (si el dueño es ec2-user)ec2-userEl directorio te pertenece, no necesitas sudo
Editar /etc/systemd/system/sudoServicios del sistema = territorio root
Ejecutar WebLogicec2-userEl dominio es de ec2-user — nunca correr WebLogic como root
chown en directorios de rootsudoSolo root puede cambiar dueños de sus propios archivos
🧠 Regla de oro de permisos

Error con "permission denied" → verifica con ls -la quién es el dueño
Dueño = root + necesito escribir → usar sudo chown
Nunca correr WebLogic o aplicaciones como root en producción

WebLogic — Dominio · Paso 4
Arrancar el Admin Server
Admin Server vs Managed Server — diferencia crítica
ℹ️ Admin Server vs Managed Server

Admin Server → el "jefe". Tiene la consola de administración. Gestiona y controla todo el dominio. Solo hay UNO por dominio.

Managed Server → los "trabajadores". Aquí corren las aplicaciones (EARs/WARs). Puede haber MUCHOS. Los controla el Admin Server.

En Credibanco: Admin Server = servidor donde entrabas a la consola web. Managed Servers = nodos SVFE donde vivían los binarios SmartVista.

# Ejecutar el script de arranque

# Esperar hasta ver esta línea:
# Server state changed to RUNNING
⚠️ El proceso bloquea la terminal

Mientras el servidor corra, esa terminal queda ocupada mostrando logs. Para administrar en paralelo abrí otra sesión SSH. En producción se usa nohup o systemctl.

Firewall de la instancia EC2 — Inbound vs Outbound
1Inbound vs Outbound
⚠️ Diferencia crítica

Inbound (entrada) → controla quién puede entrar a tu instancia. Aquí es donde agregas el puerto 7001.
Outbound (salida) → controla lo que sale de tu instancia. Por defecto All traffic y NO se modifica en la mayoría de casos.

DirecciónControlaDefault AWSModificar?
InboundQuién entra a la instanciaSolo SSH 22Sí — agregar 7001
OutboundAdónde sale la instanciaAll trafficNo tocar

4Cambiar el puerto de WebLogic (pregunta de producción)
ℹ️ Si te dicen "ese puerto no es seguro, cámbielo"

Deben cambiar en dos lugares simultáneamente o el servidor queda inaccesible:
1. Consola WebLogic: Environment → Servers → AdminServer → Configuration → General → Listen Port

Respuesta en entrevista: "El puerto se configura en WebLogic y debe reflejarse en el Security Group. En AWS además podemos restringir qué IPs acceden, lo que es más seguro que solo cambiarlo."

WebLogic — Dominio · Paso 5
systemctl — WebLogic como Servicio
Para que WebLogic arranque automáticamente cuando la instancia inicie
1¿Por qué /etc/systemd/system/?
ℹ️ El registro oficial de servicios de Linux

/etc/systemd/system/ es donde viven todos los servicios del sistema: nginx, postgresql, sshd, etc. Es el "registro oficial" de Linux moderno (systemd).

Cuando ejecutás systemctl enable weblogic, Linux busca ahí el archivo weblogic.service.

Analogía AIX: Es el equivalente de configurar el arranque en /etc/rc.d o similar. Mismo concepto, diferente implementación.

2Crear el archivo de servicio
sudo vi /etc/systemd/system/weblogic.service
ℹ️ Cómo pegar en vi

1. Abrir el archivo con vi → 2. Presionar i → modo INSERT → 3. Clic derecho en terminal = pegar → 4. ESC → 5. :wq + Enter

[Unit]
Description=WebLogic Admin Server
After=network.target

[Service]
Type=simple
User=ec2-user
# Variables de entorno que necesita WebLogic para arrancar
Environment="JAVA_HOME=/u01/jdk-17.0.12"
Environment="MW_HOME=/u01/app/oracle/middleware"
Environment="WL_HOME=/u01/app/oracle/middleware/wlserver"
# Usar /bin/sh explícito para evitar error status=203/EXEC
ExecStart=/bin/sh /u01/domains/SmartVistaDomain/bin/startWebLogic.sh
ExecStop=/bin/sh /u01/domains/SmartVistaDomain/bin/stopWebLogic.sh
Restart=on-failure

[Install]
WantedBy=multi-user.target
⚠️ Por qué /bin/sh antes del script

Sin /bin/sh systemd puede fallar con error status=203/EXEC porque no encuentra el intérprete del script. Usando la ruta absoluta de sh se garantiza que systemd pueda ejecutarlo sin depender del PATH.

3Activar y verificar el servicio
# Decirle a systemd que recargue la lista de servicios
# (obligatorio después de crear o modificar un .service)
sudo systemctl daemon-reload

# Habilitar: crea symlink para que arranque automático al boot
sudo systemctl enable weblogic

# Arrancar el servicio ahora mismo
sudo systemctl start weblogic

# Verificar estado
sudo systemctl status weblogic

WebLogic — Script de Arranque
User Data — Script Completo EC2
Script que ejecuta EC2 al crear la instancia — deja todo listo sin pasos manuales
ℹ️ Qué cubre este script

El User Data original solo instalaba herramientas básicas. Este script completo incluye todo lo que aprendiste: herramientas, LVM, swap, directorios Oracle, permisos y variables de entorno — dejando la instancia lista para instalar WebLogic sin pasos adicionales.

#!/bin/bash
# ═══════════════════════════════════════════════
# USER DATA — WebLogic 14.1.2 en EC2 RHEL 9
# Se ejecuta UNA sola vez al crear la instancia
# ═══════════════════════════════════════════════

# 1. Actualizar sistema e instalar herramientas
dnf update -y
dnf install -y lvm2 sysstat lsof unzip

# 2. Crear LVM con los dos discos adicionales
pvcreate /dev/nvme1n1 /dev/nvme2n1
vgcreate vg_weblogic /dev/nvme1n1 /dev/nvme2n1
lvcreate -n lv_domain -L 12G vg_weblogic
lvcreate -n lv_log    -L 8G  vg_weblogic

# 3. Crear filesystems y directorios de montaje
mkfs.xfs /dev/vg_weblogic/lv_domain
mkfs.xfs /dev/vg_weblogic/lv_log
mkdir /u01 /logs
mount /dev/vg_weblogic/lv_domain /u01
mount /dev/vg_weblogic/lv_log    /logs

# 4. Persistencia en fstab (usando UUID para estabilidad)
UUID_DOMAIN=$(blkid -s UUID -o value /dev/vg_weblogic/lv_domain)
UUID_LOG=$(blkid -s UUID -o value /dev/vg_weblogic/lv_log)
echo "UUID=$UUID_DOMAIN  /u01   xfs  defaults  0  0" >> /etc/fstab
echo "UUID=$UUID_LOG     /logs  xfs  defaults  0  0" >> /etc/fstab

# 5. Crear swap de 1GB (obligatorio para WebLogic)
fallocate -l 1G /swapfile
chmod 600 /swapfile
mkswap /swapfile
swapon /swapfile
echo "/swapfile  none  swap  sw  0  0" >> /etc/fstab

# 6. Crear estructura de directorios Oracle
mkdir -p /u01/app/oracle
mkdir -p /u01/domains

# 7. Dar permisos a ec2-user sobre /u01
# (sin esto WLST falla con "does not have write permission")
chown -R ec2-user:ec2-user /u01
chown -R ec2-user:ec2-user /logs

# 8. Configurar variables de entorno permanentes para ec2-user
# (se agregan al .bashrc para que carguen en cada sesión SSH)
cat >> /home/ec2-user/.bashrc << 'EOF'
export JAVA_HOME=/u01/jdk-17.0.12
export MW_HOME=/u01/app/oracle/middleware
export WL_HOME=/u01/app/oracle/middleware/wlserver
export PATH=$JAVA_HOME/bin:$MW_HOME/oracle_common/common/bin:$PATH
EOF
✅ Qué queda listo automáticamente
  • Herramientas del sistema instaladas (lvm2, sysstat, lsof, unzip)
  • LVM configurado y montado en /u01 y /logs
  • fstab con UUIDs para persistencia al reiniciar
  • Swap de 1GB activo y persistente
  • Directorios Oracle creados con permisos correctos
  • Variables de entorno listas para WLST
ℹ️ Lo que aún debes hacer manualmente después
  • Subir los instaladores (JDK 17 + WebLogic JAR) vía SFTP
  • Ejecutar el silent install de WebLogic
  • Crear el dominio con WLST
  • Configurar Security Group (puerto 7001)
  • Conectar Remote Console
  • Crear el servicio systemctl
WebLogic — Laboratorio AWS
Limpieza del Laboratorio en AWS
Orden correcto para eliminar recursos y evitar costos inesperados
⚠️ Por qué importa el orden

Borrar en orden incorrecto puede dejar recursos huérfanos que siguen generando costos aunque la instancia no exista. Por ejemplo: un Elastic IP sin instancia asociada cobra por hora. Los volúmenes EBS sin instancia también cobran.

1Orden correcto de eliminación
1
Detener y terminar la instancia EC2
EC2 → Instances → seleccionar instancia → Instance state → Terminate instance. Diferencia importante: Stop = pausa (sigue cobrando almacenamiento) · Terminate = eliminación definitiva.
ℹ️ Stop vs Terminate

Stop → instancia apagada pero los volúmenes EBS persisten y cobran.
Terminate → instancia y sus volúmenes raíz se eliminan. Usar esto para limpiar el lab.

2
Verificar y eliminar volúmenes EBS huérfanos
EC2 → Elastic Block Store → Volumes → filtrar por estado "available" (sin instancia asociada) → seleccionar → Actions → Delete volume.
⚠️ Volúmenes que pueden quedar

Los discos LAB2 y LAB3 pueden quedar en estado "available" si no se configuró eliminación automática al terminar. Siempre verificar esta sección después de terminar una instancia.

3
Liberar Elastic IP (si se creó)
EC2 → Network & Security → Elastic IPs → seleccionar la IP → Actions → Release Elastic IP address. Una Elastic IP sin instancia asociada cobra ~$0.005 por hora.
4
Eliminar Security Group personalizado (opcional)
EC2 → Security Groups → seleccionar el que creaste → Actions → Delete. No se puede eliminar si aún tiene instancias asociadas — por eso se hace al final.
5
Verificar que no quedan costos activos
AWS → Billing → Cost Explorer o ver el dashboard de facturación. También revisar en EC2 que no quede nada en estado "running" o "available".
2Recursos y su costo cuando están sin usar
RecursoCobra sin instancia?Cómo eliminar
Instancia EC2Sí (si está en Stop)Terminate (no Stop)
Volumen EBSSí — por GB/mesEC2 → Volumes → Delete
Elastic IPSí si no está asociadaRelease Elastic IP

4Relación await vs svctm — Pregunta clásica de entrevista
⚠️ Comparar siempre await contra svctm

await = tiempo en cola + tiempo de servicio

Ejemplo del laboratorio:
await = 48 ms  |  svctm = 4 ms

Conclusión: El disco físicamente NO es lento (svctm bajo). El problema es la cola de requests esperando turno para usar el disco.

🧠 Regla await vs svctm

await bajo + svctm bajo → disco saludable
await alto + svctm bajo → cola de espera, muchas operaciones compitiendo
await alto + svctm alto → disco físicamente lento (hardware degradado)

ℹ️ Pregunta de entrevista — respuesta

Si ves await alto + svctm bajo + %util alto:
El disco no tiene falla de hardware. El problema es saturación por volumen de operaciones — demasiadas escrituras compitiendo al mismo tiempo. Solución: revisar qué proceso genera tantas escrituras (iotop o lsof), evaluar agregar disco más rápido (SSD/NVMe) o redistribuir carga.

✅ Lectura SRE — las 3 herramientas juntas

top: wa = 10%, id = 60% → CPU no saturada, pero esperando disco
vmstat: b = 1, bo alto, wa = 12 → proceso bloqueado, escrituras activas
free: swap = 0, available alto → memoria OK

Diagnóstico: El problema es IO. CPU y memoria están bien. El cuello de botella es el disco. Siguiente paso: ejecutar iostat -x 1 para identificar el dispositivo y confirmar con await y %util.

🧠 Orden mental SRE para cualquier incidente

1. CPU → top: id bajo + us alto = CPU bound
2. Memoria → free: swap creciendo = memory pressure
3. IO → vmstat: wa alto + b alto = IO bound
4. Filesystem → df -h: disco lleno = app no puede escribir
5. Aplicación → si todo lo anterior está OK, el problema es la app

Siempre en ese orden. Nunca escales sin haber descartado los anteriores.

ℹ️ La diferencia clave

df (disk filesystem) → muestra la capacidad del filesystem completo: espacio usado, libre, inodes.
du (disk usage) → muestra cuánto consumen directorios o archivos específicos.

ComandoResponde aCuándo usarlo
df -h¿Qué filesystem está lleno?Primera revisión siempre
df -i¿Se agotaron los inodes?df -h muestra espacio libre pero app falla al crear archivos
du -sh /ruta/*¿Qué directorio está creciendo?Después de confirmar filesystem lleno con df
1Flujo de troubleshooting — Disco lleno
1
Ver si el filesystem está lleno
df -h
# Buscar columna Use% en 90%+ o 100%
2
Ver si los inodes están agotados
df -i
# IUse% al 100% = no se pueden crear archivos nuevos
# aunque haya espacio en disco disponible
⚠️ Trampa de inodes

El filesystem puede tener espacio libre pero si los inodes están al 100%, la aplicación falla al crear archivos con errores como No space left on device. Siempre verificar con df -i.

3
Encontrar qué directorio está creciendo
# -s = resumen | -h = human readable | * = todos los subdirectorios
du -sh /u01/weblogic/*

# Ordenar de mayor a menor para encontrar el culpable
du -sh /u01/* | sort -rh | head -10
2Causas comunes en servidores WebLogic / Java
DirectorioQué acumulaAcción
/u01/weblogic/logsLogs de aplicación y servidorRotar o archivar logs viejos
/tmp o TMPDIRArchivos temporales de JVMLimpiar temporales y verificar TMPDIR
/u01/domains/*/servers/*/logsLogs del dominio WebLogicConfigurar log rotation en la consola WLS
/var/logLogs del SO y auditoríaRevisar journald y rotar con logrotate
🧠 df vs du — "df = Foto del disco | du = Detective de directorios"

df te dice QUÉ filesystem está lleno — visión global
du te dice QUIÉN lo está llenando — visión forense

Flujo: df -h (¿lleno?) → df -i (¿inodes?) → du -sh /ruta/* | sort -rh (¿quién?)

Storage / IO — Procesos
ps aux — Estado de Procesos
Ver todos los procesos, ordenar por CPU o Memoria e identificar al culpable
1Comando base y sus flags
ps aux
# a = todos los usuarios
# u = formato orientado al usuario (muestra %CPU, %MEM, USER)
# x = incluye procesos sin terminal (daemons, servicios)
ℹ️ Diferencia con top

top muestra procesos en tiempo real actualizándose. ps aux toma una foto instantánea del momento — útil para scripting y para enviar evidencia en un incidente.

2Columnas de ps aux
USER    PID  %CPU  %MEM   VSZ    RSS  TTY  STAT  START   TIME  COMMAND
root    1234  89.0   4.2  2048M  680M  ?    R     10:01  45:12  java -jar app.jar

%MEMMemory usagePorcentaje de RAM física usadamayor a 20-30% en un proceso
VSZVirtual SizeMemoria virtual total reservadaSolo referencia — no es RAM real

3Estados STAT — columna más importante
EstadoNombreSignificaAcción
RRunningProceso ejecutándose o en cola de CPUNormal
SSleepingEsperando un evento (interruptible)Normal
DDisk waitBloqueado esperando IO — no se puede matar con kill normalInvestigar disco
ZZombieProceso terminado pero no recogido por su padreRevisar proceso padre
TStoppedProceso detenido (Ctrl+Z o señal)Verificar si es intencional
⚠️ Estado D — señal de IO bound

Si ves varios procesos en estado D simultáneamente, es confirmación directa de IO bound. El disco está tan lento que los procesos no pueden continuar. Correlacionar con iostat -x 1.

4Ordenar por CPU y Memoria
Procesos que más CPU consumen
ps aux --sort=-%cpu | head
# --sort=-%cpu = ordena de MAYOR a MENOR por %CPU
# head = muestra solo los primeros 10
Procesos que más Memoria consumen
ps aux --sort=-%mem | head
# --sort=-%mem = ordena de MAYOR a MENOR por %MEM
# El signo - antes del campo = orden descendente (mayor primero)
ℹ️ El signo menos en --sort

En --sort el signo menos (–) antes del nombre del campo significa orden descendente — del mayor al menor. Sin el menos, ordena ascendente (menor primero, no útil para encontrar el culpable).

5Comandos complementarios
# Ver proceso específico por nombre
ps aux | grep java

# Ver procesos de un usuario
ps aux | grep oracle

# Ver todos los hilos de un proceso (PID)
ps -eLf | grep PID

# Ver árbol de procesos padre/hijo
pstree -p PID
🧠 ps aux — "Foto, Ordena, Actúa"

ps aux = foto instantánea de todos los procesos
--sort=-%cpu = ¿quién consume más CPU?
--sort=-%mem = ¿quién consume más RAM?
STAT = D = proceso bloqueado en disco → IO bound confirmado

Regla: top para tiempo real · ps aux para evidencia y scripting

Glosario Técnico
Load Average
Promedio de procesos ejecutándose o esperando CPU/IO. No es porcentaje. Comparar contra número de CPUs.
CPU Idle (id)
% de tiempo que la CPU está libre. Menor a 10% = saturada. El indicador más confiable.
iowait (wa)
Tiempo que la CPU espera el disco. Mayor a 10% = IO Bound.
Memory Pressure
Sistema usa swap como RAM por falta de memoria física. Causa lentitud severa.
Swap
Espacio en disco como extensión de RAM. Disco es ~100x más lento que RAM.
CPU Bound
Procesador es el cuello de botella. id menor a 5%, us mayor a 80%.
IO Bound
Disco es el cuello de botella. wa mayor a 10%, b alto en vmstat.
LVM
Logical Volume Manager — capa entre discos y filesystem. Permite combinar discos sin reiniciar. PV → VG → LV.
XFS
Sistema de archivos alto rendimiento, predeterminado en RHEL. Recomendado para middleware.
fstab
/etc/fstab — tabla de filesystems que se montan al boot. Sin esto los LVs no persisten al reiniciar.

Instalación sin GUI usando response file (.rsp). Fundamental en servidores sin escritorio.

Reglas Mnemotécnicas
🧠 Primer paso — "CPUs Antes Que Nada"

Antes de interpretar cualquier load: nproc (Linux) o lparstat -i (AIX). Sin CPUs el load no significa nada.

🧠 3 tipos de falla — "C-M-I"

CPU Bound → id casi en cero (menor a 5%)
Memory Pressure → swap creciendo + id alto
IO Bound → wa alto (mayor a 10%) + id alto

🧠 Trampa de entrevista

Load alto NO siempre es CPU saturada. Si load alto PERO idle alto → el problema es IO o memoria.

🧠 AIX — 4 grandes cambios vs Linux

1. top → topas  |  2. systemctl → lssrc/startsrc/stopsrc
3. journalctl → errpt  |  4. df -h → df -g

🧠 LVM — "PVerde VGrande LVive"

PVerde = pvcreate | VGrande = vgcreate | LVive = lvcreate

🧠 SWAP — "FAL-CH-MK-ON"

FALlocate | CHmod 600 | MKswap | swapON

🧠 Errores WebLogic — "JDST-Display"

JDK = Oracle JDK 17 | Disco /tmp = export TMPDIR
Swap mayor a 512MB | Tinventory = crear Inventory | Display = -silent

🧠 Diagnóstico en 5 — "UP-TOP-FREE-DF-VM"

UPtime | TOP | FREE -m | DF -h | VMstat
